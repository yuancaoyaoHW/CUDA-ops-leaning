# 模拟面试脚本

## 脚本 1：CUDA 与推理系统方向（45 分钟）

### 开场（2 分钟）
面试官：请简单介绍一下你的背景和最近做的推理优化工作。

**评分标准**：能否在 2 分钟内清晰传达：浙大硕士 + vLLM 生态贡献 + 推测解码 + 性能提升数据。

---

### 第一轮：CUDA 基础（10 分钟）

**Q1**：描述 GPU 的内存层次结构，各层的延迟和带宽量级。

追问链：
- → 如果我有一个 kernel 频繁访问同一块数据，应该放在哪里？
- → Shared Memory 的 bank conflict 是什么？如何避免？
- → 如果 shared memory 不够用怎么办？L1 cache 和 shared memory 的关系是什么？

**Q2**：写一个 reduction kernel 求数组最大值，要求处理任意长度数组。

追问链：
- → 你的实现中 warp 内部可以怎么优化？（warp shuffle）
- → 如果数组有 1 亿个元素，你的 kernel 需要几次 launch？
- → 如何避免 atomic 操作的竞争？

**评分标准**：
- 5分：完整正确实现 + 所有追问回答到位
- 4分：实现正确 + 大部分追问回答好
- 3分：实现有小问题但思路正确
- 2分：只能描述思路，无法写出代码
- 1分：基础概念不清

---

### 第二轮：推理系统深度（15 分钟）

**Q3**：解释 PagedAttention 的设计动机和实现原理。

追问链：
- → 与传统 pre-allocated KV cache 相比，内存利用率提升多少？
- → Page table 的管理开销如何？对延迟有什么影响？
- → 在 beam search 场景下，PagedAttention 如何实现 KV cache 共享？

**Q4**：你实现的 EAGLE-3 推测解码，详细描述 acceptance/rejection 的过程。

追问链：
- → 为什么 rejection sampling 能保证输出分布与 target model 一致？
- → acceptance rate 受什么因素影响？如何提高？
- → 如果 draft model 和 target model 的分布差异很大，推测解码还有收益吗？
- → 你的实现中 KV cache 是如何管理的？accepted tokens 和 rejected tokens 的 KV cache 分别怎么处理？

**Q5**：vLLM 的 continuous batching 是如何工作的？

追问链：
- → 与 static batching 相比，throughput 提升的根本原因是什么？
- → 如何处理不同请求的 sequence length 差异？
- → iteration-level scheduling 和 request-level scheduling 的区别？

**评分标准**：
- 5分：原理清晰 + 实现细节准确 + 能结合自己的经验深入讨论
- 4分：原理正确 + 大部分细节准确
- 3分：高层理解正确但细节模糊
- 2分：只知道概念名称
- 1分：无法解释

---

### 第三轮：系统设计（15 分钟）

**Q6**：设计一个支持 1000 QPS 的 LLM 推理服务，模型是 70B 参数。

引导问题：
- 需要多少 GPU？如何计算？
- 用什么并行策略？TP/PP 如何选择？
- 如何处理长短请求混合的调度？
- 如何做 graceful degradation？
- 监控哪些指标？告警阈值如何设置？

**评分标准**：
- 5分：完整的系统设计 + 合理的容量规划 + 考虑了故障处理
- 4分：核心设计合理 + 大部分考虑到位
- 3分：基本架构正确但缺少细节
- 2分：只能给出粗略方案
- 1分：无法给出合理设计

---

### 收尾（3 分钟）
面试官：你有什么问题想问我的？

---

## 脚本 2：RAG 与系统设计方向（45 分钟）

### 开场（2 分钟）
面试官：介绍一下你独立负责的 RAG 系统。

**评分标准**：能否清晰描述系统架构、技术选型、核心挑战和量化成果。

---

### 第一轮：RAG 深度（15 分钟）

**Q1**：你的 RAG 系统中，chunk 策略是如何设计的？

追问链：
- → chunk size 如何确定？太大太小分别有什么问题？
- → 对于表格、代码块等结构化内容，chunk 策略有什么不同？
- → chunk 之间的 overlap 如何设置？为什么需要 overlap？

**Q2**：描述你的检索链路，从 query 到最终返回结果的完整流程。

追问链：
- → 如何处理 query 和 document 之间的语义鸿沟？（query rewriting/expansion）
- → metadata filter 是如何实现的？在向量检索之前还是之后过滤？
- → 如果检索结果质量不好，有哪些改进方向？（reranking, hybrid search）

**Q3**：RAGAS 评测具体评估了哪些维度？

追问链：
- → faithfulness 和 answer relevancy 分别衡量什么？
- → 如果 faithfulness 高但 relevancy 低，说明什么问题？
- → 如何构造评测数据集？ground truth 从哪来？

**评分标准**：
- 5分：对 RAG 全链路有深入理解 + 能讨论 trade-off + 有实际优化经验
- 4分：链路清晰 + 大部分细节准确
- 3分：基本流程正确但缺少深度
- 2分：只能描述高层架构
- 1分：无法解释细节

---

### 第二轮：向量检索与索引（10 分钟）

**Q4**：比较 HNSW、IVF、PQ 三种索引方式的优缺点。

追问链：
- → 10 亿级别的向量，用什么索引策略？
- → 如何在召回率和延迟之间做 trade-off？
- → 向量维度对检索性能的影响？降维有哪些方法？

**Q5**：如何评估和改进检索质量？

追问链：
- → Recall@K 和 MRR 分别衡量什么？
- → 如果 Recall@10 很高但 Recall@3 很低，说明什么？
- → hybrid search（dense + sparse）如何融合分数？

**评分标准**：同上

---

### 第三轮：系统设计（15 分钟）

**Q6**：设计一个企业级 RAG 系统，支持 10 万篇文档、100 并发用户。

引导问题：
- 文档处理 pipeline 如何设计？
- 索引更新策略？增量 vs 全量？
- 如何保证检索的实时性和一致性？
- 多租户隔离如何实现？
- 如何处理长文档（100+ 页 PDF）？
- 如何做 A/B 测试评估 RAG 效果？

**评分标准**：
- 5分：完整系统设计 + 考虑了可扩展性、一致性、多租户
- 4分：核心设计合理 + 大部分考虑到位
- 3分：基本架构正确但缺少细节
- 2分：只能给出粗略方案
- 1分：无法给出合理设计

---

### 收尾（3 分钟）
面试官：你有什么问题想问我的？

## 脚本 3：综合技术面（60 分钟）

### 开场（3 分钟）
面试官：请用 3 分钟介绍你最有技术深度的一个项目。

**评分标准**：叙事结构清晰、技术深度足够、有量化成果。

---

### 第一轮：项目深挖（15 分钟）

基于候选人选择的项目深入追问。

**如果选推测解码**：
- EAGLE-3 和 EAGLE-2 的区别是什么？为什么选 EAGLE-3？
- draft model 的训练数据从哪来？训练目标是什么？
- 在 NPU 上适配遇到的最大挑战是什么？如何解决的？
- 如果让你重新设计，有什么会做不同的？

**如果选 RAG 系统**：
- 系统的 QPS 和 P99 延迟是多少？
- 遇到过什么线上问题？如何排查的？
- 如果文档量增长 100 倍，架构需要怎么改？
- Research 模式和快速问答模式的技术差异在哪？

**评分标准**：
- 5分：对项目有完全的掌控力，能回答任何深度的追问
- 4分：大部分追问回答好，个别细节不确定
- 3分：高层理解好但深入追问时犹豫
- 2分：只能描述自己做的部分
- 1分：无法深入讨论

---

### 第二轮：算法与编码（15 分钟）

**Q1**：实现一个 online softmax 算法（不需要两次遍历数据）。

追问链：
- → 这个算法在 FlashAttention 中是如何使用的？
- → 为什么 FlashAttention 需要 online softmax？
- → 数值稳定性如何保证？

**Q2**：给定一个 KV cache pool 和多个请求，设计一个内存分配算法，最小化碎片。

追问链：
- → 这和操作系统的内存管理有什么相似之处？
- → PagedAttention 是如何解决这个问题的？
- → 如果要支持 prefix caching，分配策略需要怎么改？

**评分标准**：
- 5分：代码正确 + 复杂度最优 + 追问回答好
- 4分：代码正确 + 追问大部分回答好
- 3分：思路正确但代码有小问题
- 2分：只能描述思路
- 1分：无法给出方案

---

### 第三轮：分布式与系统（12 分钟）

**Q3**：Tensor Parallelism 和 Pipeline Parallelism 的区别？各自适用什么场景？

追问链：
- → TP 中 AllReduce 的通信量如何计算？
- → PP 中如何减少 bubble？（micro-batching, interleaved schedule）
- → 70B 模型在 8 卡 A100 上，你会怎么切分？

**Q4**：NCCL AllReduce 的实现原理？Ring AllReduce vs Tree AllReduce？

追问链：
- → 通信量和节点数的关系？
- → 如何 overlap 计算和通信？
- → NVLink vs PCIe 对分布式推理性能的影响？

**评分标准**：
- 5分：原理清晰 + 能做定量分析 + 有实际经验
- 4分：原理正确 + 大部分分析合理
- 3分：概念清楚但无法定量分析
- 2分：只知道名词
- 1分：无法解释

---

### 第四轮：开放讨论（12 分钟）

**Q5**：LLM 推理优化的未来方向？你最看好哪个方向？

引导：
- 推测解码的局限性和改进方向？
- 量化技术的发展趋势？
- 硬件演进对推理系统设计的影响？
- MoE 模型对推理系统的新挑战？

**Q6**：如果给你 3 个月时间和一台 8 卡 A100 机器，你会做什么项目来提升自己？

**评分标准**：
- 5分：有深度的技术洞察 + 清晰的学习路径 + 合理的项目规划
- 4分：方向正确 + 规划合理
- 3分：有想法但不够具体
- 2分：泛泛而谈
- 1分：没有想法

---

### 收尾（3 分钟）
面试官：你有什么问题想问我的？

---

| RAG 工程能力 | 15% | 全链路理解 + 有优化经验 |
| 系统设计 | 20% | 完整方案 + 考虑可扩展性和容错 |
| 编码能力 | 10% | 代码正确 + 复杂度最优 |
| 沟通与表达 | 10% | 结构清晰 + 能深入浅出 |

**总分 ≥ 4.0**：Strong Hire
**总分 3.5-4.0**：Hire
**总分 3.0-3.5**：Lean Hire（需要其他面试补充信号）
**总分 < 3.0**：No Hire
